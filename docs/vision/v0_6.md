# Canonical v0.6 Vision

`greentic-interfaces` is converging on a single canonical runtime ABI family centered on `component@0.6.0`.

Rust callsites should import the stable facade, not generated module paths:

- `crate::canonical::*` for default ABI usage
- `crate::v0_6_0::*` for explicit versioned usage

Today `crate::canonical` maps to `v0_6_0`.

## Canonical Surfaces

- `greentic:component@0.6.0`
- `greentic:types-core@0.6.0`
- `greentic:codec@0.6.0`
- `greentic:common-types@0.1.0`
- `greentic:component-v1@0.1.0`
- `greentic:pack-export-v1@0.1.0`

## Design Intent

- CBOR-first invocation path (`CallSpec`, invocation envelope)
- Stable tenant/context propagation including `i18n_id`
- Explicit descriptor/schema/runtime split for component contracts
- Additive evolution via new package versions instead of in-place breaking changes
- Internal code, tests, and README examples use canonical facade imports

## Canonical Facade (Current)

- `canonical::types` (interfaces/types shared types)
- `canonical::node` (component v0.6 node interface)
- descriptor contracts: `ComponentDescriptor`, `SchemaSource`, `IoSchema`, `SchemaRef`,
  `SetupContract`, `SetupExample`, `SetupOutput`, `SetupTemplateScaffold`, `Op`

## For New Implementations

- Prefer v0.6 + v1 surfaces only.
- Treat `0.4/0.5` component and `pack-export@0.2/0.4` as compatibility inputs, not new design targets.
- Keep provider integrations on `provider-schema-core@1.0.0` + JSON Schema.
